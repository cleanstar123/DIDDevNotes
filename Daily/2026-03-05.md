## 1. Oracle → PostgreSQL 로그인 프로시저 마이그레이션

> [!info] 배경
> - DID CMS 프로젝트의 DB를 Oracle에서 PostgreSQL로 마이그레이션하는 작업 진행
> - 로그인 기능에 사용되는 `PR_USER_LOGIN` 프로시저 변환 필요

> [!warning] 문제
> - Oracle 프로시저는 SELECT 결과가 자동으로 클라이언트에 반환됨
> - PostgreSQL에서는 함수가 결과를 반환하는 방식이 다름:
>   - `RETURNS TABLE`: 테이블 형태로 직접 반환
>   - `RETURNS refcursor`: 커서를 반환 (클라이언트가 FETCH 필요)
> - 기존 백엔드 코드에 `CUR` refcursor 파라미터가 있었으나, `CommandType.StoredProcedure` + refcursor 조합에서 파라미터 이름 대소문자 매칭 문제 발생

> [!success] 해결
> - **`RETURNS TABLE` 방식 채택** (Oracle 동작과 가장 유사)
> - `CUR` 파라미터 제거 및 `CommandType.Text`로 변경
>
> **변경 전 (Oracle):**
> ```sql
> CREATE PROCEDURE [dbo].[PR_USER_LOGIN]
> (
>   @P_USER_ID VARCHAR(50),
>   @P_USER_PW VARCHAR(200)
> )
> AS
> BEGIN
>     SELECT USER_ID, USER_NM, RESTAURANT_CODE, ...
>     FROM   CMS_USER
>     WHERE  USER_ID = @P_USER_ID AND USER_PW = @P_USER_PW;
> END;
> ```
>
> **변경 후 (PostgreSQL):**
> ```sql
> CREATE OR REPLACE FUNCTION publicdata.pr_user_login(
>     "P_USER_ID" character varying,
>     "P_USER_PW" character varying
> )
> RETURNS TABLE(
>     "USER_ID" character varying,
>     "USER_NM" character varying,
>     ...
> )
> LANGUAGE plpgsql
> AS $function$
> BEGIN
>     RETURN QUERY
>     SELECT ... FROM publicdata.cms_user CU
>     WHERE CU.user_id = "P_USER_ID" AND CU.user_pw = "P_USER_PW";
> END;
> $function$;
> ```
>
> **백엔드 코드 변경 (UserBiz.cs):**
> ```csharp
> // 변경 전
> NpgsqlParameter[] param = {
>     new NpgsqlParameter("P_USER_ID", id),
>     new NpgsqlParameter("P_USER_PW", password),
>     new NpgsqlParameter("CUR", NpgsqlDbType.Refcursor)  // 제거됨
> };
> param[param.Length - 1].Direction = ParameterDirection.Output;
> DataSet ds = PostgresHelper.ExecuteDataSet(..., CommandType.StoredProcedure, "publicdata.pr_user_login", param);
>
> // 변경 후
> NpgsqlParameter[] param = {
>     new NpgsqlParameter("P_USER_ID", id),
>     new NpgsqlParameter("P_USER_PW", password)
> };
> string sql = "SELECT * FROM publicdata.pr_user_login(@P_USER_ID, @P_USER_PW)";
> DataSet ds = PostgresHelper.ExecuteDataSet(..., CommandType.Text, sql, param);
> ```

> [!tip] 배운 점
> - **CUR 파라미터가 없어진 이유:**
>   - Oracle/SQL Server는 프로시저 내 SELECT 실행 시 결과가 자동 반환됨
>   - PostgreSQL의 `RETURNS TABLE` 방식도 동일하게 결과를 직접 반환함
>   - `refcursor` 방식에서만 커서 파라미터가 필요하며, `RETURNS TABLE` 방식에서는 불필요
>
> - **기능상 문제가 없는 이유:**
>   - 두 방식 모두 동일한 쿼리 결과를 반환
>   - 클라이언트가 받는 DataSet/DataTable 구조 동일
>   - 차이는 내부 동작 방식뿐 (refcursor: 커서 FETCH vs RETURNS TABLE: 직접 반환)
>
> - **PostgreSQL 함수 결과 반환 방식 비교:**
>   | 방식 | 설명 | 커서 파라미터 |
>   |------|------|:-------------:|
>   | `RETURNS TABLE` | 테이블 형태로 직접 반환 | 불필요 |
>   | `RETURNS refcursor` | 커서 반환 후 FETCH 필요 | 필요 |
>   | `RETURNS SETOF` | 특정 타입의 집합 반환 | 불필요 |

## 📝 메모

- `RETURNS TABLE` 방식이 Oracle 프로시저 동작과 가장 유사하여 마이그레이션에 권장됨
- `CommandType.StoredProcedure` + refcursor 조합에서는 Npgsql이 파라미터 이름을 소문자로 변환하여 대소문자 매칭 문제 발생 가능
- `CommandType.Text`로 SQL을 직접 작성하면 파라미터 매칭 문제를 피할 수 있음
